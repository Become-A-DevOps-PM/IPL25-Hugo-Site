From value proposition to value delivery

Understanding abstraction levels in software planning

It Starts with Intent

Intent is the trigger — a business need, an idea, a request. It can be vague or specific.

Intent is a value proposition: a statement of what value we want to create

Running systems are value delivery: that value actually reaching users

Everything in between — documents, diagrams, code — transforms proposition into delivery

Abstraction Levels

1. Business Requirements Document (BRD) — First formalization of intent. Captures the business need in stakeholder language.

2. Product Requirements Document (PRD) — Translates business need into product specifications: functional/non-functional requirements, user stories, constraints.

3. Architecture Documentation (C4 Model) — Defines technical design: a hierarchical view that zooms from system context down to code.

This pattern has gained traction in DevOps and the "everything as code" era

Lenses, Not Sequential Phases

These abstraction levels are lenses on the same system, not waterfall phases

Teams don't complete one level before starting the next — they cut through all levels with minimal scope

Goal: reach running software quickly, then iterate to expand

A single feature can have its own slice of business context, product spec, architecture, and code — all developed together

The C4 Model

Context — Who uses the system and how it fits into its environment

Containers — The major technical building blocks (applications, databases, servers)

Components — The internal structure of each container

Code — The actual implementation

Tech Stack Decisions

Technology stack is typically defined at the Context level — an early commitment based on PRD constraints and team capabilities

Choices are documented in Architecture Decision Records (ADRs) and guide all subsequent design

Tech stack decisions are pragmatic commitments: proceed with initial choices until concrete evidence proves them inadequate

Endless deliberation over alternatives delays progress without adding value

Probabilistic → Deterministic

Raw intent permits almost unlimited solutions

Business requirements narrow the field. Product requirements constrain it further. Architecture specifies structure.

Code is deterministic: doing exactly what it specifies, with no remaining ambiguity

But code is not the destination — running software is. Code must be deployed, configured, and operated before it delivers value.

Why Intermediate Levels Matter

Different Stakeholders, Different Altitudes — Each level enables collaboration between people who think at different altitudes

Progressive Risk Reduction — Each step down reveals risks earlier, where course correction costs less

Preserving Intent Through Implementation — Higher levels preserve reasoning behind countless micro-decisions

The Incompleteness Problem

Intermediate levels are always incomplete descriptions of the final system — this is a mathematical certainty

Planning documents represent possibilities, not certainties

Implementation reveals constraints and opportunities that no upfront planning can anticipate

Plans don't need to be complete to be useful — the value lies in direction and context, not exhaustive specification

AI as a Translation Partner

AI learns relationships between abstraction levels — how intent maps to requirements, requirements to architecture, architecture to code

Given a business need → AI drafts product requirements

Given requirements → AI proposes architectural structures

Given code → AI extracts documentation and updates higher-level artifacts

The Complementary Partnership

Humans excel downward (Abstract → Concrete): taking nebulous objectives and imagining specific features

AI excels upward (Concrete → Abstract): identifying patterns, generating documentation, proposing higher-level descriptions

Bidirectional flow: Human intent drives downward specification while AI maintains upward coherence

Together they navigate the hierarchy more effectively than either could alone

Key Takeaways

1. Intent → Running Systems
Software projects transform value propositions into value delivery through progressive formalization: BRD → PRD → C4 → Code → Running Software

2. Lenses, Not Phases
Abstraction levels are perspectives you cut through together, not sequential gates. Get to running software fast, then iterate.

3. Humans Down, AI Up
Humans drive intent into concrete specifications. AI maintains coherence upward. This complementary partnership enables faster, more aligned delivery.
