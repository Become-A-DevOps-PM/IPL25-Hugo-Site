Designing Feedback Loops

Building Software Under Uncertainty

The Core Challenge

Development must begin before complete information exists — requirements remain unclear, technical approaches are untested, user needs evolve

Traditional waterfall approaches attempt to resolve all uncertainty during planning, but comprehensive upfront analysis cannot predict the complex interactions that emerge during implementation

The alternative: design feedback loops — structured mechanisms for gathering information through implementation itself

Reframe the question from "how do we build this system?" to "what do we need to learn, and how do we design experiments to learn it efficiently?"

Minimum Viable Product (MVP)

Definition: The smallest set of functionality that provides genuine value to users — emphasis on "viable"

Purpose: Test fundamental assumptions about product-market fit. Does the core value proposition resonate? Will people actually use this?

Approach: Ruthless prioritization — features excluded not because they lack value, but because they can be deferred

Trade-off: Ships with known limitations (lower code quality, minimal error handling, adequate performance) to validate core assumptions quickly

Example: Project management app with only task creation, assignment, and status updates — no attachments, time tracking, or reporting dashboards

Walking Skeleton

Definition: A thin but complete slice through all system layers, from user interface through business logic to data persistence

Purpose: Prove that system architecture functions end-to-end — components integrate, deployment works, technical approach is sound

Approach: Execute one simple scenario completely through all layers — just enough code to prove the plumbing works

Benefit: Reveals integration problems early when cost of addressing them is minimal — authentication, serialization, network config, database connections

Limitation: Provides minimal user value initially — users cannot evaluate meaningfully until several iterations flesh out the skeleton

Rapid Prototyping

Definition: Quick, disposable experiments to explore possibilities and gather feedback — explicitly NOT production code

Purpose: Answer questions, demonstrate concepts, test assumptions — exploratory, experimental, or evaluative

Power: Low commitment enables experimentation — prototypes take hours or days, can abandon without loss

Critical Rule: Must remain clearly separated from production code — extract lessons learned and rebuild with production standards

Challenge: Maintain discipline about scope — what specific question does this prototype answer? When is it complete enough?

Spike Investigation

Definition: Time-boxed investigation of a specific technical question or unknown — produces knowledge, not demonstration

Deliverable: Brief document, recommendation, code sample, or feasibility decision — not production-ready code

Time-boxing: Fixed duration (hours to few days) prevents excessive resource consumption — stop regardless of whether definitive answer reached

Process: Focused research and experimentation — read docs, examine examples, build test cases, conduct benchmarks

Outcome: Informs subsequent work — proceed if viable, reconsider if complex, allocate more time or seek alternatives if unresolved

Strategy Selection

MVP: Do users value this solution? → Functioning product with minimal features → User adoption and feedback

Walking Skeleton: Does our architecture work end-to-end? → Complete but minimal vertical slice → Validated technical foundation

Rapid Prototyping: Is this approach worth pursuing? → Disposable demonstration → Design validation or rejection

Spike: Is this specific approach feasible? → Knowledge and recommendation → Technical decision

Projects commonly combine strategies — spikes evaluate technologies → skeleton proves integration → MVP increments validate user value

The Fundamental Insight

All strategies function as feedback loops — they produce working software or concrete investigation outcomes that provide evidence for decision-making

This evidence-based approach enables teams to fail fast on unsuccessful approaches and invest deeply in validated directions

Implementation itself is a learning activity, and designing how that learning occurs is as important as designing the software being built

Strategy selection requires identifying which uncertainties carry the most risk at the current project stage
