Designing Feedback Loops Under Uncertainty

DevOps Project Management • 2025

The Key Reframe

Stop asking: "How do we build this system?"

Start asking: "What do we need to learn, and how do we learn it efficiently?"

Development must begin before complete information exists. Attempting to resolve all uncertainty through upfront analysis does not work—complex interactions only emerge during implementation.

The alternative: design feedback loops—build strategic pieces of the system to answer specific questions and produce evidence for decision-making.

Minimum Viable Product

Core question: Do users value this solution?

The smallest set of functionality that provides genuine value. Emphasis on "viable"—users can adopt it for real work, not a demo or prototype.

Ruthless prioritization: features excluded not because they lack value, but because they can be deferred. Reach users quickly, then let feedback guide development.

Trade-off: Ships with known limitations—lower code quality, minimal error handling. Investing in polish before validating core assumptions wastes effort.

Walking Skeleton

Core question: Does our architecture work end-to-end?

A thin but complete slice through all system layers: UI → business logic → data persistence. Just enough code to prove the plumbing works.

Reveals integration problems early: authentication, serialization, network config, database connections—all become concrete when cost of fixing is minimal.

Trade-off: Minimal user value initially. Users cannot evaluate meaningfully until several iterations flesh out the skeleton with real functionality.

Rapid Prototyping

Core question: Is this approach worth pursuing?

Quick, disposable experiments—explicitly NOT production code. Exist to answer questions, then be discarded. If failure is cheap, teams explore more options.

Critical discipline: Never evolve prototypes into production code. Extract lessons learned and rebuild with production standards.

Trade-off: Prototypes answering too many questions become complex and time-consuming, losing the advantage of rapid experimentation.

Spike Investigation

Core question: Is this specific approach feasible?

Time-boxed investigation of a specific technical question. Produces knowledge—a recommendation, code sample, or feasibility decision—not a demonstration.

Time-boxing is essential: Fixed duration, then stop regardless of whether definitive answer reached. If time proves insufficient, the problem is more complex than anticipated.

Trade-off: Without time-boxing, investigations consume arbitrary time pursuing diminishing returns.

Match Strategy to Uncertainty

MVP → User value uncertain → Weeks to months → User adoption and feedback

Walking Skeleton → Architecture uncertain → Days to weeks → Validated technical foundation

Rapid Prototyping → Design options need evaluation → Hours to days → Design validation or rejection

Spike → Specific technical question → Hours to days → Technical decision

Key Takeaways

Reframe implementation as learning. Ask "what do we need to learn?" rather than "how do we build this?"

Combine strategies across project phases. Spikes → Walking Skeleton → MVP increments addresses different uncertainties at appropriate times.

Never evolve prototypes into production code. Extract lessons and rebuild with production standards.

Wrong strategy wastes effort. Validating architecture for a product users may not want is as wasteful as building features on a foundation that cannot scale.
