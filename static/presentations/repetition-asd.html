Uppgift 1 i inlämningen

Den inre utvecklingsloopen

VS Code — Utvecklarens dagliga verktyg. Kodredigering, terminalåtkomst, Git-integration, debugger.

Git — Versionshantering och samarbetsverktyg. Varje commit är en snapshot av koden.

GitHub — Central plattform för kod. Pull Requests för kodgranskning. Startpunkt för CI/CD.

Jira — Backlog-hantering, sprintplanering, uppföljning. Kopplar arbetsuppgifter till kod.

Dessa verktyg samverkar — en Jira-uppgift leder till en branch, commits, en PR, kodgranskning och merge.

Produktbackloggen

Gränssnittet mellan utvecklingsarbetet och resten av organisationen.

Källor till förändringar: nya idéer, supportärenden, buggar, teknisk skuld, marknadsföringsbehov från produktägaren.

Vårt fokus: hur förändringar hamnar i en strukturerad backlog med arbetsuppgifter som kan genomföras inom en sprint.

Från backlog → Sprint Backlog → genomförande → demo → leverans. En uppgift åt gången, från idé till "done".

Sprintplanering & Story Points

Planning Poker — uppskattar inte tid, utan komplexitet i en uppgift.

Över tid lär sig gruppen hur mycket komplexitet de klarar per sprint — detta kallas velocity.

Story points är relativa inom gruppen — de kan inte jämföras mellan olika team.

I början: dålig precision. Efter några sprintar: allt bättre uppskattningar.

Processen handlar lika mycket om kommunikation — seniora och juniora utvecklare diskuterar var komplexiteten ligger.

Git — Mer än versionshantering

Versionshantering: Spåra alla ändringar, gå tillbaka, jämföra versioner.

Samarbetsverktyg: Feature branches, Pull Requests, kodgranskning — flera utvecklare, samma kodbas.

Git log som journal: En komplett historik över utvecklingsarbetet — värdefullt för både utvecklare och AI-agenter.

Spårbarhet: Git-hashen i produktion identifierar exakt vilken version av koden som körs.

Pull Requests & Kodgranskning

Pull Request — en förfrågan om att slå samman ny kod. Inte bara merge, utan ett tillfälle för kvalitetssäkring.

Kodgranskning fångar buggar, förbättrar kodkvalitet och sprider kunskap i teamet.

Demo varje sprint — visa vad som leverats, samla feedback, anpassa nästa sprint.

Jira + GitHub-integration ger full spårbarhet från uppgift till kod till driftsatt funktion.

Everything as Code

Applikationskoden — Python/Flask i Git. Själva produkten.

Dockerfile — Beskriver hur applikationen paketeras med Gunicorn som WSGI-server.

CI/CD-pipeline — GitHub Actions YAML definierar hela leveranskedjan som kod.

Allt i Git = spårbart, granskningsbart, reproducerbart. Avgörande ur projektledarperspektiv.

Repository, SQLAlchemy

Separation of Concerns — varje lager har ett tydligt ansvar. Ändra ett lager utan att påverka de andra.

Samarbete: Olika utvecklare kan fokusera på olika lager — presentation, affärslogik eller data.

Vertical Slice: En ny feature kan implementeras genom att skära igenom alla tre lager — oberoende av andra features.

Underhållbarhet: Enklare att testa, felsöka och vidareutveckla varje del för sig.

Designmönster i applikationen

MVC (Model-View-Controller) i presentationslagret — Routes hanterar requests, Templates visar data, Models definierar struktur.

Repository Pattern i datalagret — abstraherar databasåtkomst ovanpå SQLAlchemy. Gör det enklare att byta databas.

Application Factory — Flask-appen skapas via en funktion, inte globalt. Möjliggör olika konfigurationer för dev/test/prod.

Blueprints — Organiserar Flask-routes i logiska moduler. Varje feature-område har sin egen Blueprint.

Konfiguration

Konfiguration är ett av de mest felbenägna områdena inom utveckling.

Konfigurationsklass i Python — centraliserad, typsäker hantering av miljövariabler. Inga utspridda hårdkodade värden.

12-Factor App: Konfiguration via miljövariabler — samma build, olika beteende i dev vs. produktion.

Lokalt: SQLite databas. Produktion: Azure SQL. Skillnaden styrs av en enda miljövariabel.

Hemligheter (credentials) lagras som secrets i Azure — aldrig i koden, aldrig i Git.

Säkerhet & Hemligheter

Credentials hanteras med omsorg — databaslösenord, API-nycklar och liknande ska aldrig finnas i kod eller Git.

Azure Container Apps tar emot hemligheter som krypterade miljövariabler — säkert lagrade i plattformen.

OIDC (OpenID Connect) — GitHub och Azure litar på varandra utan lösenord. Federerad identitet i CI/CD-pipelinen.

Varför OIDC? Inga lagrade credentials som kan läcka eller roteras. Kortlivade tokens istället för permanenta lösenord.

Dockerfile — receptet. Definierar exakt vad som ska finnas i containern (OS, Python, beroenden, appkod).

Docker Image — det färdiga paketet. Byggs en gång, lagras i Azure Container Registry.

Container — den körande instansen av imagen. Isolerad, portabel, identisk oavsett var den körs.

Allt styrs av kod: Dockerfile i Git → du vet exakt vad som körs i produktion.

GitHub Actions — pipelinen definieras i en YAML-fil. Automatiskt vid push till main.

Azure Container Apps uppdateras med den nya imagen — ny version körs inom sekunder.

Hela kedjan är automatiserad och reproducerbar. En push = en leverans till produktion.

Build Once, Deploy Many

En av de viktigaste principerna i 12-Factor App: bygg artefakten en enda gång.

Samma Docker-image deployeras genom: System Test → Integration Test → UAT → Stage → Produktion.

Det som skiljer miljöerna åt är konfiguration — miljövariabler pekar till olika databaser, API:er och resurser.

Fördelen: du vet att exakt samma kod som testats i stage är det som körs i produktion.

Projektledning, mognad och AI

Starta ett utvecklingsprojekt

Börja med att förstå uppgiften — men fastna inte. Gå snabbt vidare till tech stack-beslut.

Tech stack: språk, ramverk, containerisering, databas, klient, nätverk, verktyg. Formen påverkar tanken.

Börja med en senior utvecklare eller arkitekt som kan definiera en grov arkitektur. Anställ sedan fler.

Lagom teamstorlek: ~6 personer. Ratio utvecklare/testare: 3:1.

Val av tech stack

Befintlig kompetens — vad kan teamet redan? Snabbare start, lägre risk.

Målgrupp och plattform — webbapplikation, mobilapp, hårdvara? Styr val av ramverk och klient.

Community och ekosystem — populära teknologier ger mer dokumentation, fler lösningar, enklare rekrytering.

Utvecklarnas intresse — motivation att lära sig nya teknologier kan vara en faktor vid valet.

Glöm inte verktyg och infrastruktur — tillgång till internet, säkerhet på datorer, utvecklarlicenser.

Mognadsnivåer

Walking Skeleton

→

MVP

→

MMP

Walking Skeleton — minimal end-to-end-arkitektur. Bevisar att alla delar hänger ihop.

MVP (Minimum Viable Product) — minsta möjliga produkt som levererar värde och samlar feedback.

MMP (Minimum Marketable Product) — redo för marknaden. Kräver användarinstruktioner, support, driftdokumentation.

Refaktorering & Teknisk skuld

Refaktorering — göra om befintlig kod på ett arkitekturellt bättre sätt, utan att ändra funktionaliteten.

Mogna organisationer: ~30% refaktorering, ~70% nya features. En medveten balans.

Fokus på kvalitativa attribut: underhållbarhet, säkerhet, prestanda — saker som inte syns som features.

Teknisk skuld ackumuleras om refaktorering ignoreras — och blir allt dyrare att hantera över tid.

Testning & automatisering

CI/CD kräver automatiserade tester — tusentals testfall på sekunder, inte dagar.

Snabb feedback-loop: små ändringar, snabb testning, direkt leverans. Fel isoleras enkelt.

Testmiljöer längs vägen: System Test → Integration Test → UAT → Stage → Produktion.

Kort time-to-market — deploy flera gånger om dagen istället för några gånger per år.

Arkitekturdiagram

Diagram är avgörande för att kommunicera arkitektur — både inom teamet och till intressenter.

Var tydlig med perspektivet: visar du request-flödet (runtime) eller beroendestrukturen (compile-time)?

Request-flöde: Klient → Presentation → Affärslogik → Data → Databas. Visar hur ett anrop rör sig.

Beroendeperspektiv: Vilka lager som känner till vilka. Presentation beror på Affärslogik som beror på Data.

Projekt → Förvaltning

Ett projekt har en början och ett slut — men omvärlden har det inte. Någon ärver alltid resultatet.

Ratio: ~1/3 projekt, ~2/3 förvaltning. Det mesta av applikationens liv sker efter projektets slut.

Överlämning till förvaltning bör starta tidigt — för att bygga förståelse och acceptans hos mottagarna.

Tekniska lösningar påverkar organisationen — utan förankring riskerar mottagarna att vägra ta emot systemet.

AI i utvecklingsarbetet

Den inre loopen augmenteras av AI-kodagenter. Utvecklare har nu AI-kollegor som genererar kod extremt snabbt.

Tidigare: automation = CI/CD + infrastruktur. Nu: även utvecklingsarbetet accelereras med AI.

Sprint velocity ökar dramatiskt — projektledaren måste anpassa planering och förväntningar.

Det som förblir kritiskt: arkitekturförståelse och säkerhetstänkande. Utvecklare måste förstå konsekvenserna.

Molnplattformar

Vi har använt Azure — men det är bara en av många. De tre stora: Azure, AWS, Google Cloud.

Koncepten är överförbara — Container Apps, CI/CD, managed databases finns hos alla leverantörer.

Mindre alternativ: Cloudflare, Netlify, Vercel — kan vara intressanta för specifika användningsfall.

PaaS (Platform as a Service) — vi använder Container Apps och Azure SQL. Plattformen sköter infrastrukturen åt oss.
