Infrastructure as Code: Define infrastructure in text files instead of clicking through portals

Configuration as Code: Application settings in version control, not databases

Documentation as Code: Docs in the same repository as code, using Markdown

Everything as Code: Policies, security rules, and CI/CD pipelines defined as code

GitOps: Git becomes the single source of truth for all system state

WHY TREAT INFRASTRUCTURE "AS CODE"?

Version Control: Git tracks every change—who, what, when, and why

Code Review: Team members examine proposed changes before they go live

Automated Testing: Validate configurations meet security and policy requirements

Reproducibility: Create identical infrastructure from text files, not from memory

Audit Trail: Complete history of infrastructure evolution for compliance and debugging

INFRASTRUCTURE AS CODE (IaC)

Declarative Definitions: Describe desired state, not step-by-step instructions

Text Files Are Authoritative: The file defines infrastructure, not portal configurations

Student Example: Bash scripts with Azure CLI commands replace portal clicking

Cloud-Init: Define VM configuration—software, settings, services—in YAML files

Foundation Pattern: Infrastructure changes happen by editing files and reapplying

CONFIGURATION AS CODE

Application Settings: Database connections, API endpoints, feature flags in version control

Environment Parity: Development, staging, and production use the same configuration structure

Secrets Separation: Configuration files reference secrets without exposing values

Deploy Together: Configuration deploys alongside application code

Secret Management: Azure Key Vault stores sensitive values, environment variables inject them

DOCUMENTATION AS CODE

Same Repository: Documentation lives with code, written in Markdown or reStructuredText

Same Workflow: Edit files, commit, submit for review, merge when approved

Synchronized Updates: Code changes and documentation updates happen in the same pull request

Automation: Build websites or PDFs, validate links, test code examples

Living Documentation: Stays current because updating is part of the development workflow

EVERYTHING AS CODE

Policy as Code: Organizational rules and compliance as automated tests

Security as Code: Security controls embedded in development pipeline, not separate manual review

Pipeline as Code: CI/CD workflows in YAML files (GitHub Actions, GitLab CI)

Automated Validation: Static analysis, dependency checks, configuration validation run on every commit

Version Controlled Processes: Deployment procedures documented and reviewable

GITOPS: THE COMPLETE PICTURE

Single Source of Truth: Git contains complete desired state of infrastructure and applications

Continuous Reconciliation: Automated systems keep production in sync with Git repository

Change Process: To change production, commit to Git—automation applies the change

Audit and Rollback: Every change is a Git commit; rollback is a git revert

Disaster Recovery: Point new environment at Git repo, controller recreates entire system

APPLYING TO STUDENT PROJECTS

Week 1-2: Portal → Azure CLI scripts → Bash automation (IaC foundation)

Week 2-3: Cloud-init for VM configuration, Flask config files in Git

Week 3-4: Markdown documentation in repository, validation scripts for infrastructure

GitOps Principles: Use Git as single source, make changes through version control

Progressive Learning: Understand the spectrum, practice fundamental techniques

KEY TAKEAWAYS

Progression, Not Checklist: Move along the spectrum based on project needs and maturity

Engineering Practices: Code review, testing, CI/CD apply to infrastructure and operations

Text Files Enable: Version control, reproducibility, automation, and collaboration

Start Simple: Scripts and cloud-init provide significant benefits without complexity

Foundation Building: Understanding the philosophy prepares you for sophisticated approaches
