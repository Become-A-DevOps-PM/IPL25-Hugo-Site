Navigating the abstraction hierarchy in software planning

Understanding probabilistic vs deterministic representations

The Abstraction Hierarchy

1. Business Intent - Strategic objectives and value propositions

2. Requirements Specification - Stakeholder needs and system constraints

3. Product Requirements Document (PRD) - Feature definitions and success criteria

4. User Stories and Epics - User-centric functionality descriptions

5. Backlog Items - Specific, prioritized work units

The Abstraction Hierarchy (continued)

6. Definition of Done - Acceptance criteria and completion standards

7. Code - Executable implementation (the only deterministic level)

Each level provides greater specificity than its predecessor while remaining incomplete until code executes

Nature of Abstraction Levels

Probabilistic Nature (Levels 1-6): Higher levels operate in the realm of possibility and probability—they leave implementation space open

Deterministic Reality (Level 7): Code transitions from probabilistic to deterministic—it produces specific, measurable behavior

"Code is Truth": Running code is the only artifact that unambiguously describes what the system actually does

This incompleteness is not a flaw but a fundamental characteristic of the planning process

Why Intermediate Levels Matter

1. Shared Understanding Across Stakeholders - Different groups engage at different levels, each bringing specialized knowledge that applies most effectively at specific altitudes

2. Progressive Risk Reduction - Each step down the hierarchy reduces uncertainty, revealing risks at stages where course correction costs less

3. Managing Complexity Through Decomposition - Breaks large systems into tractable units without requiring anyone to hold the entire system in mind

4. Preserving Intent Through Implementation - Maintains the reasoning thread from strategic vision through thousands of micro-decisions

AI as a Translation Layer

Pattern Recognition: AI systems learn relationships between abstraction levels from vast software artifacts

Bidirectional Translation: AI can generate requirements from intent, or extract documentation from code—working in both directions

Probabilistic Output: AI translations represent possibilities, not certainties—they provide starting points that accelerate translation but require validation

Contextual Intelligence: AI with organizational knowledge generates proposals aligned with existing systems, architecture, and team capabilities

Human-AI Complementarity

AI Excels Upward (Concrete → Abstract): Given code, AI identifies patterns, extracts functionality, generates documentation—remarkable at recognizing what implementations represent

Humans Excel Downward (Abstract → Concrete): From nebulous objectives, humans imagine specific features, envision experiences, exercise judgment about what matters

Complementary Partnership: Humans drive intent into specification; AI generates drafts and maintains alignment; together they navigate the hierarchy more effectively

Bidirectional Feedback Loop: Human intent drives downward while AI maintains upward coherence—creating alignment neither could achieve alone

Practical Implications for PM

Perfection Less Critical: Since higher levels are inherently probabilistic, investing effort to eliminate all ambiguity wastes resources

Iteration Cycles Shorten: Translation between levels accelerates—moving from intent to backlog in hours rather than weeks

Documentation Maintenance Automated: AI propagates changes through abstraction levels, keeping specifications aligned with reality

Tighter Strategic Coupling: AI-assisted translation maintains connection between business vision and daily implementation

Key Takeaway

The Progression: Software moves through abstraction levels from business intent to executable code

The Reality: Each level operates probabilistically until code produces deterministic behavior

The Collaboration: AI translates between levels (excelling upward) while humans guide direction (excelling downward)

The Outcome: Faster iteration, better alignment, and more sustainable planning practices across the abstraction hierarchy
